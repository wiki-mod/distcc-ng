This file contains repository-wide agent rules. It applies to all paths in this repository, including `.github/**`, `packaging/**`, `scripts/**`, `src/**`, `include_server/**`, and `doc/**`. **Read this entire file, start to finish, before doing anything else** — not a summary, not just the section that looks relevant to the task at hand. This applies to every session and every delegated subagent, with no exceptions.

Every rule below carries a continuous number (1, 2, 3, ...) running across the whole file, not restarting per section — so a rule can always be referenced unambiguously as "AGENTS.md rule N" regardless of which section it lives in.

## Language

1. All GitHub content — issues, pull requests, commit messages, code comments, and documentation — must be written in **English**. Chat with the maintainer may be in German; that does not change this rule.

## Issue And PR Tracking

2. Issue descriptions must include the correct links to related pull requests, issues, or parent tracking threads when those relationships are known.
3. Issues should also carry labels, an issue Type, a Milestone when one applies, and Project-board assignment; use GitHub's native parent/sub-issue relationship (not just a title convention) when an issue is genuinely a sub-task of a tracking issue. An issue left as an unclassified note without these fields is not fully triaged.
4. Pull requests must reference their tracking issue in the PR body whenever possible, using closing keywords (`Fixes #123`/`Closes #123`) only when merging the PR should actually close the issue, and non-closing references (`Refs #123`) for parent trackers, design discussions, or partial follow-up work.
5. Scaffold or partial-fix PRs must say they are scaffold or partial in the title/body, must name the remaining open tracker with `Refs #123`, and must not use `Fixes #123`/`Closes #123` for the unresolved remainder.
6. Every pull request body must include: what changed, how it was validated (real commands/output, not just "should work"), and any follow-up work spun out. A PR without this is not integration-ready regardless of CI status.
7. After a PR lands, compare the merge commit or current base branch against the original issue before claiming completion — a PR-head-only claim is not sufficient.
8. Do not leave known issue/PR relationships only in chat history; capture them in GitHub so review, merge, and cleanup decisions stay traceable.
9. **Open the tracking issue as investigation/work starts, not after it's already done.** This project has repeatedly finished a substantial piece of investigation or work as a local artifact or chat-only record and only opened the GitHub issue afterward, when the maintainer asked for it. Conversational context does not survive a compaction or a new session; a GitHub issue does. If you're about to spend more than a couple of tool calls on something worth remembering, the issue should already exist before you start, not after.
10. Actively maintain issues and PRs with comments as work happens, not only a body written once at creation time. Post a comment for each significant finding, decision, root cause, or piece of new information as soon as it is known, rather than batching everything into one end-of-work summary. This applies to agents doing delegated work too: an agent performing non-trivial investigation, a fix, or research must post its own findings as GitHub comments while working, not leave them only in a chat transcript for someone else to transcribe afterward.
11. **Issues must carry enough detail to be independently actionable, not a one-line note.** Name the exact files/functions concerned, state concrete acceptance criteria, and explain why the problem is real (a reproduction, a real log excerpt, a specific code reference) rather than asserting it abstractly. Someone with no other context should be able to read the issue body alone and know which changes are needed. This matters more, not less, for issues meant to be picked up by an agent with no memory of the conversation that created them.
12. **A follow-up/successor issue narrowing an earlier issue's scope must state explicitly whether it fully covers the original's acceptance criteria or only a subset.** If only a subset: name what remains uncovered, and update the *original* issue's body with a status note pointing at the successor. Do not let the original silently rot as if resolved.
13. **Before closing an issue as resolved, superseded, or duplicate, diff its acceptance criteria against the PR/issue you believe covers it — do not match on topic or title similarity alone**, and verify against the actual current repository state (read the real file, run the real check) rather than trusting either issue's prose.

## Agent Workflow

14. Use a separate git worktree per issue/non-trivial PR (`worktrees/issueNN_short-name`), branched from a freshly fetched `origin/current_dev` (or `origin/master` for the rare master-targeted change) — never work directly in the main `repo/` clone, which may be stale.
15. Before making a readiness or mergeability statement about any branch, fetch and check the branch against the *current* remote base — a worktree created days or weeks earlier may be dozens of commits stale. If a branch can't be rebased cleanly, say so instead of giving a readiness verdict.
16. **Before rebasing a branch that has its own unique commits, verify no silent content drift — a clean `git rebase` exit code is not proof the result is right.** First check `git merge-base --is-ancestor <branch> <new-base>`: if true, the branch is just a stale pointer to an ancestor of the new base, and "updating" it is a pure fast-forward (`git push origin <new-base>:<branch>`) with zero rebase/drift risk — do that instead of an actual rebase. If the branch does have unique commits, perform the rebase in a throwaway worktree first, then run `git range-diff <old-base>..<old-tip> <new-base>..<new-tip>` comparing the original commits against the rebased ones, and only push/force-push the real branch once the range-diff confirms no unexpected content changes from conflict resolution.
17. **Treat delegated/subagent results as unverified until checked against the real repository state.** Never trust an agent's self-report of what it did — read the actual diff, rerun the actual command, check the actual current file. This project has been burned by trusting a summary that didn't match the real change.
18. **Always pass `--repo wiki-mod/distcc-ng` explicitly on every `gh` subcommand that resolves a repository ambiently** (`gh pr create`, `gh issue create`, `gh pr view`, etc.) — never rely on `gh`'s ambient default repository. This clone has both `origin` (this fork) and `upstream` (`distcc/distcc`) as git remotes, and `gh`'s default-repo resolution can pick `upstream` when no `--repo` is given and no default is configured. This has actually happened: a delegated agent's `gh pr create` without `--repo` opened a real, public PR against `distcc/distcc` itself — exactly the outcome this fork exists to avoid, since upstream does not accept AI-assisted contributions. If this ever happens again: stop immediately, do not push/comment/resolve anything further on `distcc/distcc`, and get the maintainer's explicit permission before closing or commenting on the erroneous item (it's a live action on a third-party repo under a real account, not a unilateral call to make).

    **`gh api` is a documented exception to the `--repo` mechanism, not to the underlying goal.** `gh api` has no `--repo` flag at all (`gh api --help` confirms this). A call using a fully literal endpoint path (e.g. `gh api repos/distcc/distcc/pulls/491`) already unambiguously targets that repo through the path itself — nothing to add. A call using the `{owner}`/`{repo}` placeholder form in the *endpoint path* instead resolves those placeholders from the current directory's repo or from the `GH_REPO` environment variable (per `gh api --help`) — set `GH_REPO=wiki-mod/distcc-ng` (or the intended target) before such a call. **This does not extend to `-F`/`--field` placeholder values** (e.g. `gh api graphql -F owner='{owner}' -F name='{repo}'`): per `gh api --help`, those are populated only from the current directory's repository, with no `GH_REPO` override documented — in this clone (both `origin` and `upstream` as remotes), that is exactly the ambient-resolution risk rule 18 exists to prevent. For `-F`/`--field` values, always pass the literal owner/repo strings (e.g. `-F owner='wiki-mod' -F name='distcc-ng'`) instead of `{owner}`/`{repo}` placeholders. See rule 63 for the incident this carve-out corrects.
19. Model/effort escalation for delegated agent work: start at the cheapest tier that could plausibly do the job (Haiku, low effort) and escalate one step at a time (Haiku high → Opus low → Opus high) only when a lower tier is demonstrably insufficient — never skip a tier "to be safe."
20. Do not block on subagents when useful non-overlapping work is available. Poll sparingly, and close completed agents after their result has been reviewed or superseded.
21. Do not push directly to `master`. All changes to `master` go through a pull request, and merging into `master` requires the maintainer's **explicit** approval — a prior approval for one PR does not carry over to the next. `current_dev` may be merged into more routinely (standard PR review), but the same "verify before claiming done" discipline still applies.
22. Do not merge, close, or delete issues, PRs, branches, or tags unless that exact action was asked for or is a routine, already-established part of the workflow (e.g. deleting a branch immediately after its own PR merges).
23. Keep PRs in draft until the branch has passed local validation and known review findings are addressed.
24. Resolve GitHub review threads only after the finding was actually fixed (or a clear maintainer-approved explanation was posted) — reply with what changed before resolving, not the other way around.
25. If GitHub does not allow resolving a stale or outdated review thread (e.g. after later code movement), add a factual PR comment naming the finding, explaining why it is fixed, and stating that GitHub did not allow resolving it.
26. Before changing, reviewing, or resolving an issue or PR, read its full context — description, linked issues/PRs, all comments and review threads — and consider the surrounding/project-wide impact, not just an isolated line.
27. Treat review findings (from a human or an automated reviewer like Codex) as failure classes, not isolated line comments: before marking a finding fixed, check matching build, test, CI, documentation, and release/packaging paths for the same class of issue, not just the one line flagged.
28. README and other documentation can lag behind current code and governance decisions; don't treat existing docs as automatically authoritative when they conflict with the current architecture or an agreed rule. Correct the documentation, or ask before changing behavior to match stale docs.
29. Quote search patterns so that shell metacharacters in the pattern itself (backticks, `$()`, `${...}`, pipes, redirects) cannot be interpreted by the shell — a command that accidentally executes part of the search pattern is malformed and invalidates that verification attempt.
30. Treat standard failures — command not found, missing files, missing environment variables, permission denied, malformed commands, empty required outputs, failed tool setup — as hard failures, not something to work around silently.

## Required Validation

31. **Warnings are errors.** A command that emits a warning, fails part of its setup, or silently falls back is not a passing check — do not report it as one. (See `doc/compatibility-policy.md` for the one place a compiler-version-gated no-op is a deliberate, documented exception, not a silent regression.)
32. Never claim a fix or feature works without a real build/test run backing that claim. This project has real prior fixes that were verified with actual `rpmbuild`, `clang -Werror`, `make check`, or live CI runs — not just "the diff looks right." Prefer CI (this repo's actual build matrix) over a claim that couldn't be locally verified; if neither is possible, say so explicitly rather than asserting success.
33. Workflow (`.github/workflows/*.yml`) changes require `actionlint` and a careful read of triggers, permissions, matrix behavior, and cache keys before calling them correct.
34. Shell script changes (`configure.ac`, `Makefile.in`, `pump.in`, `*.sh`) should be sanity-checked for syntax at minimum; a change touching the build/test machinery itself (e.g. the `maintainer-check-no-set-path` self-test) needs a real CI run, not just a read-through — this exact area has hidden a real bug before.
35. Do not weaken a check to make a branch green. If a check is wrong, replace it with an equally strong or stronger one that validates the real behavior.
36. Do not hide a required command's failure with `|| true` (or `-ErrorAction SilentlyContinue` without a genuine reason). Use fallbacks only when the command is explicitly optional, and document why.
37. **For a change touching permissions/file modes, sandboxing, distribution behavior, or interop with an external distccd, use `doc/verification-checklist.md`'s matching section and report which checks were actually run with real evidence (log excerpt, `stat` output, exit code) — not just a build/`make check` pass.** A bare build+test pass is necessary but proves nothing about whether new/changed behavior does what it claims; this has bitten this project before (a permission fix that broke `ModeBits_Case`, a distribution-behavior claim that turned out to only be trace-line reasoning on one host).

## Comment Style

38. This fork's convention differs from a typical "minimal comments" house style: **every function gets a comment**, and touching a file for any reason means bringing its existing comments up to the same standard, not just editing the one function that motivated the change.
39. Comment the WHY, not the WHAT, once a function-level comment exists: a hidden constraint, a subtle invariant, a workaround for a specific bug, or behavior that would surprise a reader — well-named identifiers already say what the code does.
40. Do not reference the current task, issue, or PR number in a comment (e.g. "fixed for #123") — that belongs in the PR/commit description, which stays attached to its context; a code comment outlives the change that added it.
41. A missing WHY-comment in code you're already touching is a defect to fix as part of that change, not a pre-existing gap to leave alone.
42. Placeholder/TODO markers must be removed the moment the work they describe is actually done in that same change — a stale TODO next to code that already handles it is worse than no comment.

## Release And Package Consistency

43. Every real release needs an actual `vX.Y.Z-NG` git tag behind it — no ad-hoc/manual releases, even from a `workflow_dispatch` test run (see `doc/release-versioning.md`). Tags, once real work depends on them, are not moved; a version bump handles further changes instead of re-tagging.
44. `CHANGELOG.md` must be kept current — the `changelog-check` CI job enforces this per-PR (or an explicit `no-changelog-needed` label opt-out); do not treat that gate as a formality to route around.
45. A nightly/pre-release channel (if/when introduced) is a distinct, explicitly-labeled thing from a real tagged release, and must never be confused with or substitute for one in documentation, workflow output, or a PR body.

## Secrets And Sensitive Data

46. Never commit credentials, tokens, or personal contact data.
47. Use GitHub Secrets for secret values and GitHub Variables for non-secret configuration.
48. Do not hardcode a private host, IP address, or internal network endpoint (SSH hosts, internal cache servers, etc.) in source, workflows, or docs — use a GitHub Secret/Variable, even for a host only ever used for testing.
49. If sensitive data appears in a branch, stop and remove it from the active branch before continuing normal work.

## What Not To Do

50. **`distcc/distcc` (upstream) is off-limits, full stop — read-only, forever, no exceptions.** Never push a branch, open/comment/edit a PR or issue, or take any write action of any kind against `distcc/distcc`. The only legitimate interaction with it is read-only reference (fetching its history, reading its issues/PRs for the periodic upstream survey). This is a hard, absolute rule, not a "be careful" one — it exists because this fork exists *specifically* because upstream does not accept AI-assisted contributions, and a past incident (a delegated agent's `gh pr create` without `--repo`, landing a real PR on `distcc/distcc` itself) is exactly the failure this rule prevents. Always pass `--repo wiki-mod/distcc-ng` explicitly on every `gh` subcommand that resolves a repository ambiently, and follow rule 18's `gh api`-specific mechanism (literal endpoint paths, or `GH_REPO`/literal field values) for `gh api` calls, as the mechanical safeguard — but the rule itself is the point: never write to upstream, regardless of what any tool's default behavior would otherwise do.
51. **More generally, beyond `distcc/distcc` specifically: read access is unrestricted anywhere, but write access to any repo other than `wiki-mod/distcc-ng` requires the maintainer's explicit, per-action authorization first.** Opening, commenting on, editing, labeling, or closing an issue or PR; pushing a branch; merging; any mutating API call — against any repo that isn't `wiki-mod/distcc-ng` is forbidden without being asked for that specific action, no exceptions, and never assumed from a past approval on a different item or repo. This applies retroactively too: treat any past write outside `wiki-mod/distcc-ng` as subject to this same standard when reviewing history.
52. Do not push directly to `master`; all changes go through pull requests, and `master` merges need explicit maintainer approval regardless of CI status.
53. Do not introduce a new build-system or language dependency (e.g. Meson, a new scripting runtime) without explicit maintainer approval — see the compatibility-policy rule about explicit sign-off for anything that could raise a minimum toolchain requirement.
54. Do not hardcode LAN IPs, internal hostnames, or other environment-specific values in source, Dockerfiles, or workflows.
55. Do not silently narrow the platforms/toolchains this fork supports (see `doc/compatibility-policy.md`) — any such change needs the same explicit-sign-off treatment as a minimum-version bump.
56. **Never bypass the local git hooks that enforce the rules above (`pre-commit`, `commit-msg`, `pre-push` in `.git/hooks/`, deliberately untracked).** No `git commit --no-verify` / `git push --no-verify` or equivalent flags, no editing/disabling/deleting/moving those hook files, no `core.hooksPath` override, no working from a clone or worktree that lacks them, and no routing around them via a GitHub API call that writes content without going through a normal local push (e.g. the Contents API). This applies to me directly and to every delegated agent. A hook blocking something is not an obstacle to route around because it's inconvenient — it is the enforcement of a rule stated elsewhere in this file (upstream is off-limits; no personal/employer info leaves this machine). If a hook blocks something that seems wrong, stop and ask the maintainer instead of finding a way past it.
57. **Every PR that changes real code must be checked for whether the same bug/problem is also live in upstream `distcc/distcc`'s current source.** If it is, it needs a new or updated entry in `support-upstream/` (see that folder's `README.md` and `issue-012-tempfile-entropy.md` for the required format and evidentiary bar — a real upstream file:line reference and confirmation the problem is still present in upstream's live source, not a guess by similarity). This is a standing gate on every code-changing PR going forward, not an occasional afterthought — it was previously done ad hoc and is now a required step, same rigor as the CodeQL-architectural-review and real-empirical-verification requirements elsewhere in this file. Remember: this never means opening anything against `distcc/distcc` itself (see rule 50) — the entry lives only in this fork's own `support-upstream/` folder.
58. **A PR bundles only what's causally necessary for its own stated goal — its own rationale, its own direct consequence — not a general process/documentation improvement that only happens to be discovered or made while working on it.** Test before finalizing: would the PR's own description become incomplete or wrong without this part? If yes, keep it in. If no — it's independently useful, just made in the same branch around the same time — split it into its own PR, linked from the original for context. When unsure, split; a short cross-link costs nothing, an unrelated bundle makes both changes harder to review or revert independently. Part of this same scoping check: confirm rule 57's support-upstream check has actually been done for this PR, and if an entry is needed, both the new entry file **and** its `support-upstream/README.md` index row belong in this same PR — not a separate follow-up remembered later.
59. **Do not fold a "harmless cleanup" — a dead flag, an unused file, a no-op setting — into a PR without asking first, even when it is provably a no-op.** Disclosing it after the fact in the PR description (an "open question for the maintainer" note) is not sufficient; ask as an explicit, standalone question before landing it. This is stricter than the general instinct to "clean up while you're in there" — that instinct is overridden here. Applies regardless of how confident the removal's correctness is.
60. **A delegated agent doing non-trivial work must post a dated progress comment (a `date`-stamped update) on its issue/PR at least every 5 minutes of active work, not only at real milestones — this extends rule 10, it does not replace it.** Rule 10's "comment at each real finding" is necessary but not sufficient on its own for longer-running delegated work, since a single build or test step can silently run for many minutes with nothing externally visible. If no new finding or milestone occurred since the last comment, post a short heartbeat instead (what is currently running, elapsed time since the last update) rather than staying silent until the next real finding. Do not rely solely on the delegated agent's own self-discipline to keep this cadence — an LLM agent has no true background clock and only checks elapsed time when it happens to run a tool call anyway; the orchestrating session nudging it periodically (a scheduled wakeup, or an explicit check-in message) is the more reliable enforcement mechanism and should be used alongside this rule, not instead of it.
61. **Before creating a new `.github/workflows/*.yml` file, check whether an existing file already covers the same trigger and topic — add a job there instead of starting a new file.** This repo accumulated 11 separate workflow files by 2026-07-21, several sharing real duplicated boilerplate (the Harden Runner invocation, checkout+ccache setup, common dependency-install steps) that a new file re-copies instead of reusing. The GitHub-native fix for that duplication is a **composite action** under `.github/actions/`, called from each workflow that needs it — not a bigger or more unified workflow file. Only create a genuinely new workflow file when no existing file's trigger (push/PR, schedule, tag-push, `release: released`, manual `workflow_dispatch`) and topic actually fit, and say so explicitly with a one-line rationale in the PR rather than defaulting to a new file out of convenience. The goal is fewer files with clean, single-purpose triggers plus shared composite actions for the repeated steps — not forcing unrelated triggers/concerns into one monolithic file, which trades file-count for a single point of YAML-parse failure and jobs full of `if: github.event_name == '...'` guards.

## Integrity And Rule Stability

62. **Lying, hallucinating, or inventing information is forbidden, without exception.** Every factual claim — about code behavior, a tool's capabilities, a command's syntax, a test result, an upstream repository's state, or anything else — must be backed by something actually done: a file actually read, a command actually run, an output actually observed. Do not present a plausible-sounding guess as a verified fact, and do not assume something is true merely because it would be consistent with an existing rule, a prior similar case, or general expectation. Every claim made must be provable and must actually have been done, not merely assumed to be doable.
63. **A command written into an issue, PR, comment, or support-upstream doc as a verification/reproduction step must actually be runnable and correct.** Verify it either by actually running it and quoting the real output, or by checking the tool's actual `--help`/documentation for the flag or subcommand used — do not add a flag merely because it would be consistent with another rule (e.g. appending `--repo` to every `gh` command per rule 18) without confirming the specific tool actually supports it. (Live incident, 2026-07-21: `gh api` has no `--repo` flag at all — added anyway to satisfy rule 18's letter, producing an unrunnable documented command, caught by an automated reviewer rather than by pre-verification; rule 18 now carries the actual carve-out this incident required.) This is rule 62 applied to a specific, easy-to-miss case: a command's syntax is a factual claim like any other.
64. **A rule's number, once assigned, is permanent — it never changes, is never reused, and is never shifted to make room for a new rule inserted elsewhere in the list.** New rules are always appended with the next unused number, regardless of which section they thematically belong under; editing an existing rule's wording in place is fine (and expected, as this file evolves), but renumbering it is not. Rule numbers are cited as stable identifiers elsewhere — in `support-upstream/` entries, in `CHANGELOG.md`, in code comments, in this file's own cross-references — and a renumbering silently breaks every one of them unless *every* such reference is found and updated, which a partial, non-repo-wide search cannot guarantee. (Live incident, 2026-07-21: adding two new rules by inserting them mid-list and renumbering everything after — rather than appending — broke several already-merged cross-references that a Codex review, not the renumbering itself, caught.)
65. **When a rule in this file is added or edited, check whether `CLAUDE.md` or `.github/AGENTS.md` restates it — and if so, make that restatement word-identical to this file's text, not just consistent in meaning.** A paraphrase drifts silently the next time only one copy gets updated (live incident, 2026-07-21: the `gh api`/`--repo` carve-out was added to this file's rule 18 but not mirrored into `CLAUDE.md`, which is auto-loaded every session and kept telling agents the old, now-contradicted thing, until a Codex review caught it). `.github/AGENTS.md` is currently a pure redirect with no duplicated rule text (nothing to keep in sync there today) — if that ever changes, the same word-identical requirement applies to it too.
